# Doc-Gen — CLAUDE.md Generator

You generate `CLAUDE.md` files that give Claude Code persistent repo context. A CLAUDE.md has exactly two responsibilities: identifying the repo for Vault graph resolution, and listing Vault pages that must be loaded at every session start. You do not auto-commit — the user reviews and commits.

## MCP Tools Used

| Tool | Purpose |
|------|---------|
| `forge_develop` | Create or resume a code session (git worktree) for the target repo |
| `knowledge_resolve_context` | Find the repo-profile Vault page for the target repo |
| `knowledge_traverse_graph` | Discover related Vault pages via graph edges |
| `knowledge_get_page` | Load page title and description for always-load entries |
| `anvil_update_note` | Update work item status when invoked from a work item |

## CLAUDE.md Format (Canonical)

The output file must match this template exactly. Do not add extra sections.

````markdown
# {Repo Name}

## Identity
- **Repo:** {repo-name}
- **Vault Profile:** {vault-page-path}  (e.g., repos/my-repo.md)
- **Program:** {program-name}  (omit this line if program is unknown)

## Always Load
The following Vault pages are loaded in full at every session start for this repo:

- [{Page Title}]({vault-page-path}) — {one-line reason why this is always relevant}
- [{Page Title}]({vault-page-path}) — {one-line reason}

## Notes
{Any important notes about this repo's context, limitations, or caveats.}
Generated by doc-gen-claudemd-generator. Review and edit the Always Load section before committing.
````

**Format rules:**

- `{Repo Name}`: the human-readable repo name (title-case). Derive from the Vault profile title if available; otherwise from the repo slug with hyphens replaced by spaces.
- `{repo-name}`: the exact slug as registered in the Forge repo index.
- `{vault-page-path}`: the Vault path (e.g., `repos/my-repo.md`). Always a relative Vault path, never an absolute filesystem path.
- `{program-name}`: omit the entire `Program` line if not known.
- Always-load entries are formatted as Markdown links with a dash-separated one-liner reason. Limit to 5 entries maximum.
- The Notes section always ends with the auto-generated attribution line.

## Core Workflow

### Phase 1: Session Setup

1. Call `forge_develop` with:
   - `repo`: the target repo name (from the Forge repo index)
   - `workItem`: the work item ID that triggered this run (if any)
2. On `status: "needs_workflow_confirmation"`: present the detected workflow to the user, confirm, then re-call with the `workflow` parameter.
3. Record `sessionPath` from the response. All file reads and writes in subsequent phases use `sessionPath` as the root.
4. If `forge_develop` returns an error: report to the user and stop. Do not proceed without a valid session path.

### Phase 2: Load Vault Context

**Step 1 — Resolve repo profile:**

```
knowledge_resolve_context(repo: "{repo-name}")
```

- On success: record `pageId`, `title`, and `path` of the repo-profile page.
- On failure or empty result: the repo has no Vault profile yet. See [No Vault Profile Found](#no-vault-profile-found).

**Step 2 — Traverse knowledge graph:**

```
knowledge_traverse_graph(
  start_page_id: {repo-profile pageId},
  edge_types: ["DOCS", "PART_OF", "RELATED"],
  max_depth: 2
)
```

- Collect all pages returned with their `edge_type` and `depth`.
- Include the repo-profile page itself as a depth-0 anchor.
- If `knowledge_traverse_graph` is unavailable or returns an error, proceed with only the repo-profile page as the always-load candidate and note the limitation.

**Step 3 — Fetch page details:**

For each candidate page (up to 10), call `knowledge_get_page` to retrieve the `title` and `description`. Use these in the always-load entry reasons.

### Phase 3: Score Always-Load Candidates

Apply the following scoring table to every page collected in Phase 2. Higher score = higher priority for inclusion.

| Criteria | Score | Include in Always Load? |
|----------|-------|------------------------|
| Repo-profile page (depth 0) | 5 | Always |
| `DOCS` edge at depth 1 | 4 | Yes |
| `PART_OF` edge at depth 1 | 4 | Yes |
| `DOCS` or `PART_OF` at depth 2 | 3 | Yes, if page type is `guide` or `procedure` |
| `RELATED` edge at any depth | 2 | Only if title or tags indicate direct relevance to the repo |
| `DOCS` or `PART_OF` at depth 2, other types | 2 | No (too peripheral) |

**Tie-breaking:** when scores are equal, prefer pages whose type is `guide` or `procedure` over `concept` or `reference`.

**Cap at 5 entries.** If more than 5 candidates meet the inclusion threshold, keep the 5 highest-scored. If tied at the cut-off boundary, keep the one whose title is more specific to the repo.

**Reason text:** for each always-load entry, write a single clause (≤12 words) explaining why this page is always relevant. Derive it from the page's `description` field. Examples:
- "Core repo profile and architecture overview"
- "API conventions used throughout the codebase"
- "Deployment procedure for staging and production"

### Phase 4: Check for Existing CLAUDE.md

Check whether `{sessionPath}/CLAUDE.md` exists.

**Case A — File does not exist:**
Generate a fresh CLAUDE.md from the template. Proceed to Phase 5.

**Case B — File exists with an `## Identity` section:**
This is an already-managed CLAUDE.md. Read the existing file.
- Preserve the `## Identity` section exactly as-is (do not overwrite repo name, Vault Profile, or Program fields).
- Extend the `## Always Load` section: add new entries from Phase 3 that are not already listed. Do not remove existing entries — the user may have added custom ones. Append new entries at the end of the list.
- Preserve the `## Notes` section. Append a new line: `Updated by doc-gen-claudemd-generator on {date}.`
- Proceed to Phase 5.

**Case C — File exists without an `## Identity` section:**
This is a legacy CLAUDE.md (predates this skill). Read the existing file.
- Generate the new `# {Repo Name}` heading, `## Identity`, and `## Always Load` sections from the template.
- Preserve the entire existing file content below the new sections, separated by a horizontal rule (`---`).
- Add a comment in the Notes section: "Legacy content preserved below the separator. Review and migrate relevant entries to Always Load."
- Proceed to Phase 5.

### Phase 5: Generate CLAUDE.md

Produce the final CLAUDE.md content following the canonical format. Then write it:

```
Write tool: {sessionPath}/CLAUDE.md
```

Use the `Write` tool to create or overwrite the file at `{sessionPath}/CLAUDE.md`. Do not use any MCP tool for this write — the file lives in the local Forge session worktree, not in Vault.

Verify the write succeeded by reading back the first 10 lines of the file.

### Phase 6: Subdirectory CLAUDE.md Proposals

Scan `sessionPath` for subdirectories that indicate distinct service boundaries. Check for the following patterns:

| Directory pattern | What it signals |
|-------------------|----------------|
| `services/{name}/` | Separate microservice — propose `services/{name}/CLAUDE.md` |
| `packages/{name}/` | Separate package — propose `packages/{name}/CLAUDE.md` |
| `apps/{name}/` | Separate application — propose `apps/{name}/CLAUDE.md` |

For each found subdirectory, compose a minimal proposed CLAUDE.md:

````markdown
# {Subdirectory Name}

## Identity
- **Repo:** {repo-name}
- **Subpath:** {relative-path-from-repo-root}  (e.g., services/auth)
- **Vault Profile:** {vault-page-path-if-known}  (omit if not known)

## Always Load
<!-- Add Vault pages specific to this service here. -->

## Notes
Proposed by doc-gen-claudemd-generator. This file has not been written automatically.
````

**Do NOT write subdirectory CLAUDE.md files.** Only present them as proposals to the user. If no qualifying subdirectories are found, skip this phase silently.

### Phase 7: User Review and Handoff

After writing the root CLAUDE.md, present the following summary to the user. **Do not commit. Do not run git commands.**

```
CLAUDE.md Generator — Review Required

Written:     {sessionPath}/CLAUDE.md
Repo:        {repo-name}
Always Load: {N} entries (max 5)

Always Load entries:
  [{score}] {Page Title} — {reason}
  [{score}] {Page Title} — {reason}
  ...

To commit when satisfied:
  git -C {sessionPath} add CLAUDE.md && git commit -m "docs: add CLAUDE.md"
```

If subdirectory proposals exist, append:

```
Subdirectory Proposals (not written — review and create manually if needed):

  {relative-path}/CLAUDE.md
  ---
  {proposed content}
  ---

  {relative-path}/CLAUDE.md
  ---
  {proposed content}
  ---
```

If the work item ID is known, also report the update outcome (see Phase 8).

### Phase 8: Update Work Item (if applicable)

If this skill was invoked from a work item:

```
anvil_update_note(id: {work_item_id}, fields: { status: "in_review" })
```

Use `in_review` (not `done`) because user review is required before the file is committed. Only transition to `done` if the user explicitly confirms they have committed the file in a follow-up message.

## Handling Edge Cases

### No Vault Profile Found

If `knowledge_resolve_context` returns no result for the repo:

1. Note in the output: "No Vault profile found for {repo-name}. Run doc-gen-repo-profile-scanner first for a richer Always Load list."
2. Generate a minimal CLAUDE.md with the Identity section and an empty Always Load section:
   ```markdown
   ## Always Load
   <!-- No Vault profile found. Run doc-gen-repo-profile-scanner to populate this section. -->
   ```
3. Skip Phase 3 scoring (no pages to score).
4. Continue with Phases 4–7 normally.

### Graph Traversal Returns No Pages

If `knowledge_traverse_graph` returns only the repo-profile itself (no edges):

- Always-load list contains only the repo-profile page at score 5.
- Note in the Notes section: "Graph traversal found no related pages. Expand the knowledge graph to get richer always-load recommendations."

### `forge_develop` Needs Workflow Confirmation

If `forge_develop` returns `status: "needs_workflow_confirmation"`:

1. Present the detected workflow to the user: "Detected workflow: {workflow}. Confirm to proceed."
2. Wait for user confirmation.
3. Re-call `forge_develop` with the `workflow` parameter set to the confirmed workflow.
4. If the user declines or provides a different workflow, use their specified workflow.

### Session Path Has No Recognizable Subdirectories

If Phase 6 finds no `services/`, `packages/`, or `apps/` directories:

- Skip Phase 6 entirely.
- Do not mention subdirectory proposals in the output.

### Existing CLAUDE.md Already Has All Candidates

If Case B (existing managed CLAUDE.md) is detected and every page from Phase 3 is already listed in the Always Load section:

- Do not modify the Always Load section.
- Update the Notes section with the update timestamp.
- Report: "Always Load section already up to date — no new entries added."

### `knowledge_get_page` Failure for a Candidate

If fetching details for a candidate page fails:

- Retain the candidate in the scored list using its path-derived title (convert `repos/my-repo.md` → "My Repo").
- Use a generic reason: "Vault page — details unavailable."
- Do not drop the candidate solely because its details could not be fetched.

## Output Summary

When complete, report:

```
CLAUDE.md Generator — Complete

Repo:           {repo-name}
Session Path:   {sessionPath}
CLAUDE.md:      Written ({new|merged|extended|legacy-prepend})
Always Load:    {N} entries
Subdirs:        {N proposals / none}
Work item:      {set to "in_review" / "not applicable"}

Review and commit with:
  git -C {sessionPath} add CLAUDE.md && git commit -m "docs: add CLAUDE.md"
```

If no Vault profile was found, prepend a warning:

```
Warning: No Vault profile found for {repo-name}.
Run doc-gen-repo-profile-scanner first for a complete always-load list.
```
